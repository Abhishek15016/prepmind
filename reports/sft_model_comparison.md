## Evaluation Criteria

Each answer is judged on: **correctness, domain accuracy, clarity, safety, helpfulness,
less generic response, and better domain-specific behavior.**

| 1 | What is the difference between LoRA and QLoRA? | LoRA adds small trainable matrices instead of updating all weights; QLoRA uses quantization to reduce memory. | LoRA freezes the base model entirely and trains small low-rank update matrices (A and B) injected into attention/MLP layers - only a few million parameters move instead of billions. QLoRA additionally loads the frozen base in 4-bit NF4 precision while keeping the LoRA adapters and compute in bf16, plus uses double quantization and paged optimizers to avoid OOM crashes. Interview takeaway: LoRA cuts *trainable* parameters; QLoRA additionally cuts the *base model's memory footprint* - together they let you fine-tune a 7B+ model on a single free-tier GPU. | Fine-Tuned | Names the actual mechanism (NF4, double quantization, paged optimizers) and gives the practical "why it matters" framing an interviewer expects, not just a definition. |
| 2 | Why does self-attention need positional encoding? | Self-attention doesn't understand word order, so positional encoding adds position information. | Self-attention is permutation-invariant by construction - it would score "dog bites man" and "man bites dog" identically without help. Positional encoding fixes this by injecting order information into the embeddings (sinusoidal) or by rotating Query/Key vectors by an angle proportional to position (RoPE), which modern models prefer because the resulting dot product encodes *relative* distance and generalizes better to longer sequences than seen in training. | Fine-Tuned | Explains *why* permutation-invariance is a problem, and distinguishes RoPE from absolute encoding with a concrete production reason (long-context generalization). |
| 3 | What is Retrieval-Augmented Generation and why does it reduce hallucination? | RAG combines a language model with retrieval to generate more accurate, grounded answers. | LLMs hallucinate because their factual knowledge is compressed, static parameters formed at training time - when a query needs a fact outside that compressed knowledge (or the model is simply uncertain), it still produces a fluent-sounding guess. RAG fixes this by fetching relevant source passages at query time and conditioning generation on them, so the model has to compose an answer from text actually in front of it rather than purely from memory - it also lets you cite sources, which retraining a bigger model never gives you. | Fine-Tuned | Explains the causal mechanism behind hallucination and why grounding (not just "more accuracy") is the actual cure, plus a concrete production benefit (citations). |
| 4 | Explain the difference between an LLM agent and a simple prompted LLM call. | An agent can use tools and make decisions over multiple steps; a simple call returns one response. | A single prompted call is a snapshot: prompt in, one response out, no loop. An agent wraps the LLM in an iterative reasoning loop (think, act via a tool call, observe the result, decide the next step) that continues until the task is done or a stopping condition is hit - the defining feature is *dynamic control flow the model itself decides*, not just tool access. A pipeline that always calls the same 3 tools in a fixed order isn't really agentic, even with tools; a system that decides *which* tool, *when*, and *whether to loop again* is. | Fine-Tuned | Draws the actual dividing line (model-decided control flow vs fixed pipeline) that base answer gestures at but never states precisely. |
| 5 | What is DPO, and how is it different from RLHF with PPO? | DPO aligns models with human preferences without RL; RLHF with PPO trains a reward model then uses RL. | RLHF/PPO needs four models in memory (policy, frozen reference, reward model, critic) and an unstable on-policy RL loop. DPO re-derives the RLHF objective's closed-form optimal policy and substitutes it directly into the loss, so you train only the policy model on (prompt, chosen, rejected) triples using its own log-probabilities relative to a frozen reference - no reward model, no PPO rollouts, no critic. It behaves like supervised learning on preference pairs while optimizing the same underlying objective as RLHF. | Fine-Tuned | Explains the actual substitution mathematically at a plain-English level and names the concrete engineering cost RLHF/PPO carries that DPO removes. |
| 6 | Why do we chunk documents before embedding them for RAG? | Embedding models have limited input length, and smaller chunks improve retrieval precision. | Chunk size is a precision/recall tradeoff: chunks too large dilute the embedding with multiple unrelated ideas, so a query about one specific fact retrieves a chunk where that fact is drowned out by everything else in it; chunks too small lose the surrounding context needed to make sense of the fact at all. The highest-impact fix isn't shrinking chunk size blindly, it's cutting at *natural meaning boundaries* (headings, paragraphs) with some overlap, so no chunk is missing the context it needs to stand alone as a retrievable unit. | Fine-Tuned | Names the actual tradeoff mechanism (dilution vs missing context) rather than only citing an embedding input-length constraint. |
| 7 | What is the KV cache and why does it speed up LLM inference? | It stores key/value vectors from previous tokens so they aren't recomputed each step. | Without caching, generating token 500 would require recomputing Key/Value vectors for all 499 prior tokens from scratch - purely wasted work since those vectors never change once computed. The KV cache stores them after the first pass, so each new token only computes its own K/V and reuses the rest - this is exactly why time-to-first-token (full prefill) and time-per-token (cheap, cached) behave so differently in production, and why the cache's growing memory footprint is what serving frameworks like vLLM specifically optimize with PagedAttention. | Fine-Tuned | Connects the mechanism to a concrete production consequence (prefill vs decode latency split) instead of stating the fact in isolation. |
| 8 | What is Model Context Protocol (MCP) and what problem does it solve? | Uncertain - "may refer to a protocol for connecting AI models to external context or tools." | MCP is an open standard that lets any LLM application connect to any tool/data source through one common interface, instead of every app writing a custom integration for every tool (the "M x N integration problem"). It defines three roles - host (the LLM app), client (maintains the connection), and server (exposes tools, resources, and prompts) - so a tool built once as an MCP server works with any MCP-compatible host, the same way USB-C standardized device connectors instead of every device needing its own cable. | Fine-Tuned | The base model visibly hedges/doesn't know; the fine-tuned model gives a confident, structurally correct, domain-accurate answer - the clearest before/after gap in this table. |
| 9 | How would you evaluate a RAG system end-to-end? | Check relevance of retrieved documents and quality of the answer, e.g. via accuracy or human review. | Retrieval and generation must be scored *separately* because either can fail independently - perfect retrieval with a generator that ignores the context still produces a wrong answer, and a great generator fed irrelevant chunks still fails, so a single end-to-end score hides which half broke. Track retrieval with context precision/recall against a labeled query-document set, and generation with faithfulness (is the answer actually supported by the retrieved context, catching hallucination even when retrieval was correct) and answer relevancy (does it address the actual question) - frameworks like RAGAS operationalize exactly this split. | Fine-Tuned | Gives concrete, named metrics and explains *why* separating retrieval/generation scoring matters, not just "check quality." |
| 10 | What is catastrophic forgetting, and why is LoRA less prone to it than full fine-tuning? | It's losing previously learned abilities after fine-tuning; LoRA "may reduce this somewhat." | Catastrophic forgetting happens because gradient updates needed to specialize on new data can overwrite the same weights that encoded prior, unrelated capabilities. Full fine-tuning lets every parameter move, so a narrow dataset can drift the model far from its original broadly-capable weights. LoRA structurally bounds this: the base weights stay entirely frozen, only small low-rank adapters train and get added on top - the model's original knowledge is preserved by construction, and the adapter can even be removed to instantly restore base behavior. | Fine-Tuned | Explains the actual mechanism (frozen weights as a structural safeguard) instead of the hedged "may reduce this somewhat" in the base answer. |
